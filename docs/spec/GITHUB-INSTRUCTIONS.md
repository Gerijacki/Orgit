# Orgit — SEO and Positioning Guidelines for GitHub

## Goal

Orgit's goal is not only to be a good tool.

It must become a highly visible project within the GitHub ecosystem and appear recurrently in:

- GitHub Trending
- GitHub search results
- Social media shares
- Development newsletters
- Technical blogs
- Videos and demos

The whole repository must be designed to make it easy for a visitor to go from "seeing the project" to "leaving a star" in less than a minute.

---

# Philosophy

Every element of the repository must answer one question:

> Does it make someone understand the project faster?

If the answer is no, it must be simplified.

Clarity always takes priority over the amount of information.

---

# Positioning

Orgit must be positioned as:

> **The Autonomous Repository Evolution Engine for Claude Code.**

It must never be sold simply as a refactorer.

Its value proposition is much greater.

It must convey that it is capable of:

- understanding projects
- detecting problems
- planning improvements
- executing changes
- evolving complete repositories

# README

The README must be the project's main marketing element.

It is not a technical manual.

It is a landing page.

In less than thirty seconds it must answer:

- What it does.
- Why it is different.
- How it is used.
- What results it produces.

---

# Recommended README order

## Hero

Name.

Description.

Badges.

Main image.

---

## GIF

The first thing the visitor should see is a real demo.

It must show the complete cycle.

Analyze.

Plan.

Improve.

Validate.

Finalize.

No explanations.

Just the result.

---

## Problem

Explain what problem it solves.

Don't talk about the project.

Talk about the developer's pain.

---

## Solution

Show how Orgit removes that problem.

---

## Example

Show the complete flow.

It must be extremely short.

The visitor should think:

"I want to try this."

---

## Result

Show what the user gets.

Don't explain how it works internally.

Show only the benefit.

---

## Features

Explain the features.

Always from the user's point of view.

Not from the technical point of view.

---

## Use cases

Show real situations.

For example:

- Legacy project
- Open source project
- Monorepo
- Startup
- Large team

---

## Roadmap

Show only the important features.

It must convey that the project is alive.

---

## Contributing

Make collaborating as easy as possible.

# Psychological goal

When someone arrives at the repository they should think three things:

"This solves a real problem."

"I want to try it."

"I'm going to give it a star to follow its evolution."

The whole repository design must aim to provoke exactly that reaction.

---

# Vision

Orgit must not present itself as just another AI tool.

It must position itself as the reference for the autonomous evolution of repositories through Claude Code.

The goal is that, when a developer thinks about improving, modernizing, or maintaining a software project with the help of AI, Orgit is the first name that comes to mind.
